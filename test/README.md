# 项目测试说明

这个目录放的是项目的自动化测试，用来检查外卖系统的主要功能链路是否还能正常工作。

## 怎么运行

在项目根目录运行：

```powershell
.\.venv\Scripts\python.exe manage.py test test -v 2
```

项目根目录指的是这个目录：

```text
C:\Users\20674\Documents\food_deliver
```

测试运行时，Django 会自动创建一个临时测试数据库：

```text
test_the_food_mas2
```

测试结束后，这个测试数据库会被自动删除。它不会修改你真正导入数据的数据库：

```text
the_food_mas2
```

## 测试原理

这些测试不是靠人工打开网页点击，而是用 Django 自带的测试工具模拟用户操作。

主要原理如下：

1. 使用 `django.test.TestCase`

   Django 会为每次测试准备独立的测试数据库。测试里创建的用户、订单、商品等数据都只存在于测试数据库里，不会污染真实数据库。

2. 使用 `django.test.Client`

   `Client` 可以模拟浏览器请求。例如：

   - `client.get("/food/")` 模拟打开美食列表页
   - `client.post("/foodorder/?foodid=1", {...})` 模拟用户提交点单表单
   - `client.delete("/blog/delete/?blogid=1")` 模拟删除博客

3. 使用 session 模拟登录

   项目当前是通过 `request.session["user_id"]` 判断登录用户的，所以测试里会直接写入 session，模拟普通用户、骑手、商家、管理员已经登录。

4. 使用 ORM 检查数据库结果

   请求发出后，测试会用模型查询数据库，确认业务状态真的改变了。例如：

   - 用户点单后，检查 `Order` 是否创建
   - 骑手接单后，检查订单 `pos` 是否从 `0` 变成 `1`
   - 商家备餐后，检查订单 `pos` 是否变成 `2`
   - 用户评价后，检查订单评分和状态是否保存
   - 酒店和娱乐场所评价后，检查平均评分是否更新

5. 使用 mock 隔离外部 AI 服务

   AI 助手接口会调用外部大模型服务。测试时不会真的请求外网，而是用 mock 替代外部调用，只检查：

   - 接口能正常响应
   - 菜品、酒店、娱乐信息会放进系统提示词
   - 返回 JSON 格式正确

这样可以保证测试稳定，不受网络或第三方服务影响。

### 美食订单链路

完整覆盖了：

```text
用户下单 -> 待骑手接单 -> 骑手接单 -> 商家备餐 -> 骑手取餐 -> 骑手送达 -> 用户评价
```

对应订单状态：

```text
0 -> 1 -> 2 -> 3 -> 4 -> 5
```

### 跨端互动

测试中单独覆盖了“同一个订单在不同角色端的显示和响应”：

- 用户下单后，骑手端 `/rider/` 能看到待接订单，包括菜品、顾客、地址和“接单”按钮。
- 骑手接单后，商家端 `/space/` 能看到当前订单，包括菜品、顾客、骑手、地址和“完成备餐”按钮。
- 商家备餐后，骑手端 `/space/` 能看到订单进入可取餐状态，并显示“已取餐”操作。
- 骑手取餐后，骑手端 `/space/` 能看到订单进入配送中状态，并显示“已送达”操作。
- 骑手送达后，用户端 `/space/` 能看到订单进入可评价状态，并显示“去评价”操作。

也就是说，测试不是只检查数据库里的订单状态变化，还会检查对应角色页面是否能看到应该出现的订单和按钮。

## 没有自动覆盖的内容

这些内容目前没有自动测试，主要因为它们更适合人工检查或浏览器端测试：

- 页面视觉效果、CSS 样式、布局是否好看
- 浏览器里真实点击、滚动、弹窗体验
- 真实外部 AI 服务是否可用
- 接近 2 MB 的图片上传边界情况
- 不同浏览器和不同屏幕尺寸下的显示效果

目前这套测试主要覆盖的是后端请求、数据库状态变化、权限判断和主要业务流程。

## 测试文件说明

- `test_database_config.py`

  检查数据库配置、SQL dump 中的数据库名、MySQL 核心表是否存在。

- `test_project_workflows.py`

  覆盖项目主要功能流程，包括用户端、骑手端、商家端、管理员端、食品订单、酒店、娱乐、博客和 AI 助手。
